## 一、行为差异与规范一致性概览

### 问题 D：声明位变体标注 `in`/`out` 与 Java use-site 差异 ⭐⭐⭐ HIGH

**用例：** GEN_05_01_301~325 系列

**实际行为：**
```typescript
// ArkTS 使用声明位变体标注
interface Producer<out T> {
  produce(): T
}
interface Consumer<in T> {
  consume(x: T): void
}

// Java 使用 use-site 通配符
// Producer<? extends T>, Consumer<? super T>
```

**对比：**
| 维度 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 变体机制 | 声明位 `in`/`out` | use-site `? extends`/`? super` | 集合协变 | 结构类型系统 |
| 协变返回值 | ✅ `out T` 允许 | ✅ `? extends T` | ✅ | ✅ |
| 逆变参数 | ✅ `in T` 允许 | ✅ `? super T` | ✅ | ✅ |
| 不变 | ✅ 默认 | ✅ 默认 | ✅ `var` 参数 | ✅ |
| 只读字段协变 | ✅ `out T` + readonly | ✅ `? extends` | ✅ `let` | ✅ |
| 回调嵌套变体 | ⚠️ Spec 要求检查但编译器未完全实现 | N/A（无声明位变体） | ✅ 正确检查 | N/A |
| 构造函数中变体 | ✅ out 允许 | N/A | ✅ | N/A |

**影响：**
- 声明位变体更简洁，但需要设计师在定义时考虑变体
- Java 的 use-site 更灵活，但使用更繁琐
- 回调嵌套变体检查待完善

**建议：** 完善回调嵌套变体检查的实现。
