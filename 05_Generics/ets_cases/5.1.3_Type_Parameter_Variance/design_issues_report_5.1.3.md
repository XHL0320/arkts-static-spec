## 一、行为差异与规范一致性概览

### 问题 D：回调嵌套变体检查未递归 ⭐⭐⭐ HIGH

**用例：** GEN_05_01_307~310

**实际行为：**
```typescript
interface Producer<out T> {
  produce(): T
}
interface Consumer<in T> {
  consume(x: T): void
}
// 嵌套回调函数的变体检查未被递归检测
```

**Spec 依据：** §5.1.3 Type Parameter Variance — 变体检查应递归到嵌套类型内部

**对比：**
| 语言 | 回调嵌套变体检查 | 说明 |
|------|---------------|------|
| ArkTS | ⚠️ Spec 要求检查但编译器未完全实现 | es2panda 未递归展开回调类型 |
| Java | N/A（无声明位变体） | 机制不同 |
| Swift | ✅ 正确检查 | 与 ArkTS Spec 要求一致 |
| TypeScript | N/A（结构类型系统） | 不适用变体标注 |

**影响：** 
- 嵌套变体违规未被检测，可能导致运行时类型不安全
- 与 Spec §5.1.3 要求不一致

**建议：** 完善回调嵌套变体检查的实现，递归展开回调函数类型内部。

## 二、语言设计差异

### 差异 D：声明位变体标注 `in`/`out` 与 Java use-site 差异 ⭐⭐⭐ HIGH

**说明：** ArkTS 使用声明位变体标注（`in`/`out`），Java 使用 use-site 通配符（`? extends`/`? super`）。这是语言设计层面的选择差异，非缺陷。ArkTS 的声明位变体更简洁，Java 的 use-site 更灵活但使用更繁琐。

**对比：**
| 维度 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 变体机制 | 声明位 `in`/`out` | use-site `? extends`/`? super` | 集合协变 | 结构类型系统 |
| 协变返回值 | ✅ `out T` 允许 | ✅ `? extends T` | ✅ | ✅ |
| 逆变参数 | ✅ `in T` 允许 | ✅ `? super T` | ✅ | ✅ |
| 不变 | ✅ 默认 | ✅ 默认 | ✅ `var` 参数 | ✅ |
| 只读字段协变 | ✅ `out T` + readonly | ✅ `? extends` | ✅ `let` | ✅ |
| 构造函数中变体 | ✅ out 允许 | N/A | ✅ | N/A |
