# 5.2 Variance and Wildcard - 跨语言对比报告 ArkTS / Java / Swift

## 概览

Variance（变体）和 Wildcard（通配符）定义了泛型类型参数的协变/逆变规则。ArkTS 采用声明点变体（declaration-site variance），与 Swift 一致；Java 则使用使用点变体（use-site variance / wildcards）。

## 章节对应关系

| ArkTS (§5.2) | Java (JLS §4.5) | Swift (Generics) |
|------|------|-------|
| 协变 `out` | `? extends T` | 协变泛型 |
| 逆变 `in` | `? super T` | 逆变泛型（不常见） |
| 不变 `inout` | 默认（无通配符） | 默认 |
| 通配符 `*` | `?` | `some`/`any` |
| 声明点变体 | ❌（使用点） | ✅ |

## 关键差异矩阵

| 维度 | ArkTS | Java | Swift |
|------|-------|------|-------|
| 协变 out | ✅ | ✅ `? extends` | ✅ |
| 逆变 in | ✅ | ✅ `? super` | ✅ |
| 通配符 | `*`（推断时） | `?` | `some`/`any` |
| 声明点变体 | ✅ | ❌（使用点） | ✅ |
| 变体位置检查 | ✅ 编译期 | ✅ 编译期 | ✅ 编译期 |

## 用例 1:1 对照

### 声明点协变
| ArkTS | Java | Swift |
|-------|------|-------|
| `class C<out T> { ... }` | ❌（无法声明点标注） | `class C<out T> { ... }` |

### 使用点通配符
| ArkTS | Java | Swift |
|-------|------|-------|
| 不支持使用点通配符 | `List<? extends Number>` | `[some Number]` |

| 语言 | 是否报错 | 说明 |

## 综合评分

| 维度 | ArkTS | Java | Swift |
|------|:----:|:----:|:-----:|
| 变体表达力 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 类型安全性 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 使用复杂度 | ⭐⭐⭐⭐⭐（声明点更简单） | ⭐⭐（使用点更复杂） | ⭐⭐⭐⭐⭐ |
| 通配符灵活性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

## 核心结论

ArkTS 采用声明点变体是正确选择——比 Java 的使用点通配符更简单直观，且与 Swift 一致。当前编译器变体检查粒度止于成员签名表层（未递归检查回调函数类型内部），已在 issue_report 记录为 GAP。

## ArkTS 设计建议

编译器应增强变体检查的递归深度，确保回调函数类型参数内部的变体违规也被检测。这与 spec 要求一致，属实现层面 bug。
