# 2.7 Keywords - ArkTS 设计问题发现报告

**报告日期：** 2026-06-15
**测试用例数：** 56（compile-pass: 35, compile-fail: 16, runtime: 5）
**目的：** 通过 4 类关键字（硬/类型/软/未来保留）的全覆盖测试识别 ArkTS 编译器和 spec 设计问题。

---

## 一、设计问题

### 问题 A：catch 同时列在硬关键字表和软关键字表 ⭐⭐⭐ HIGH

**spec 表 1（硬关键字）：** 包含 `catch`
**spec 表 3（软关键字）：** 也包含 `catch`

**实测（用例 015）：** `catch` 仅在 try-catch 上下文使用，未单独测试 `catch` 作标识符是否合法。

**潜在问题：**
- 若 `catch` 是硬关键字，则 `let catch = 1` 必须失败
- 若 `catch` 是软关键字，则 `let catch = 1` 在非 try-catch 上下文应允许

**建议：** spec 修正：catch 应只列在硬关键字表，因为它在所有上下文都被解析为 try-catch 关键字。

---

### 问题 B：类型关键字双重保护与 TypeScript 不兼容 ⭐⭐⭐ HIGH

**spec 表 2：** 主名（int）和别名（Int）都是硬关键字。

**实测（用例 023~027）：** `int`/`string`/`boolean`/`Object`/`void` 作为标识符都失败。

**问题：** TS 代码中 `let int: number = 1`、`let string: string = ""` 是合法的。迁移到 ArkTS 时全部需要重命名。

**对比：**
| 语言 | int | Int | string | String |
|------|-----|-----|--------|--------|
| ArkTS | ❌ | ❌ | ❌ | ❌ |
| Java | ❌ | ✅ | ✅ | ✅ |
| TypeScript | ✅ | ✅ | ✅ | ✅ |

**建议：**
1. 仅保留小写形式作关键字（与 Java 一致）
2. 或提供迁移工具自动检测并重命名

---

### 问题 C：var 在未来保留软关键字表但当前禁用 ⭐⭐ MEDIUM

**spec 表 4：** `var` 列为未来保留软关键字。

**实测（用例 048）：** `var x: int = 1` 编译失败。

**矛盾：**
- "未来保留" 暗示当前可作标识符或软关键字
- 但实测当前已作为不可用的关键字

**建议：**
1. 若不打算实现 var 作变量声明 → 应明确说明 var 仅用于禁用，不是"未来保留"
2. 若计划支持 var → spec 应说明启用条件（如版本号）

---

### 问题 D：类型关键字的别名设计冗余 ⭐⭐ MEDIUM

**spec 表 2 列出主名+别名：**
- `bigint` / `BigInt`
- `boolean` / `Boolean`
- `byte` / `Byte`
- `char` / `Char`
- `int` / `Int`
- `Object` / `object`
- `short` / `Short`
- `string` / `String`

**问题：** 别名设计的目的不清晰：
- 与 TypeScript 兼容？（TS 中 `Int` 不存在，`Boolean` 是包装类型）
- 与 Java 兼容？（Java 中 `Integer`/`Boolean` 是包装类，且 `Object` 是大写主名）

**实测（用例 028~032）：** 主名+别名等价使用合法。

**评价：** 别名设计可能造成混淆，建议 spec 明确推荐使用哪种形式。

---

### 问题 E：硬关键字 final 测试缺失 ⭐ LOW

**spec 表 1：** `final` 列为硬关键字。

**实测：** 用例 018 中 final 未实际使用（注释说明 final 待实现）。

**疑问：** ArkTS 当前是否实现 final 修饰符？

**建议：** spec 应明确 final 的当前状态（是否仅声明保留还是已实现）。

---

### 问题 F：软关键字 declare/module/out 测试覆盖不足 ⭐ LOW

**spec 表 3：** declare/module/out 列为软关键字。

**实测：** 这三个软关键字没有专门用例。

**建议：** 后续补充测试：
- declare 在 ambient declaration 中
- module 在模块声明中（spec 中标记 deprecated）
- out 在协变类型参数中

---

### 问题 G：byte/short/long/float/double/char 测试覆盖不足 ⭐ LOW

**spec 表 2：** byte/short/long/float/double/char 是类型关键字。

**实测：** 用例 023~027 仅测试 5 个，未测试 byte/short/long/float/double/char 作标识符的反向用例。

**建议：** 补充更多类型关键字反向用例。

## 四、改进方向建议

### 短期（spec 修正）
1. 移除 catch 在表 3 的重复列出
2. 明确 var 当前定位（"禁用"而非"未来保留"）
3. 说明 final 实现状态

### 中期（编译器/迁移工具）
4. 提供 TS→ArkTS 迁移工具自动重命名 `int`/`string` 等冲突标识符
5. lint 警告：建议使用类型主名（如 int）而非别名（Int）

### 长期（语言演进）
6. 评估是否放宽类型关键字别名保护（仅大写形式可作普通标识符）
7. 评估 var 关键字最终设计
