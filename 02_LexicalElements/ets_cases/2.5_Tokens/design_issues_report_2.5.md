# 2.5 Tokens - ArkTS 设计问题发现报告

**报告日期：** 2026-06-15
**测试用例数：** 35（compile-pass: 26, compile-fail: 4, runtime: 5）
**目的：** 通过 4 类 Token 的全面测试识别 ArkTS 编译器和 spec 的设计问题。

---

## 一、设计问题

### 问题 A：Spec 未列出完整的 Operator 和 Punctuator Token 表 ⭐⭐ MEDIUM

**spec 描述：** §2.5 仅声明 4 类 Token 存在，具体的 Operator/Punctuator 集合需要查看 §2.8。

**问题：** 实际开发中需要查阅多个章节才能确认某符号是否合法 Token，例如：
- `===` 是否为 Token？（§2.5 未列出）
- `??` 是否为 Token？（§2.5 未列出）
- `=>` 是否为 Token？（§2.5 未列出）

**对比：**
| 语言 | Token 列表完整性 |
|------|---------------|
| ArkTS | ⚠️ §2.5 概览，§2.8 详细 |
| Java JLS §3.5 | ✅ 集中列出 |
| ECMAScript §12 | ✅ 集中列出 |

**评价：** 章节划分合理但增加查阅成本。

**建议：** §2.5 增加 Operator 和 Punctuator 的完整速查表，详细规则保留在 §2.8。

---

### 问题 B：最长匹配原则与 char 字面量交互未充分定义 ⭐⭐ MEDIUM

**spec 原文：** "The next token is always the longest sequence of characters that form a valid token."

**测试场景：** char 字面量 `c'A'` 在 tokenization 时如何处理？
- 是先识别 `c` 为标识符再尝试合并？
- 还是直接识别为 char 字面量前缀？

**实测：** 编译器正确识别 `c'A'` 为单一 char 字面量 Token（用例 LEX_02_01_007 已验证）。

**疑问：** 当遇到 `let cVar = c'A'` 时，`c` 后跟标识符部分（如 `var`），是否会因最长匹配优先识别为 `cvar` 标识符？

**对比：** 此场景 Java/Swift 无对应（无 `c'...'` 语法）。

**评价：** 当前实现行为正确，但 spec 应明确 char 字面量在最长匹配中的优先级。

---

### 问题 C：装饰器 `@` 字符的非 Token 使用未明确 ⭐ LOW

**实测（用例 028）：** `let x: int = @` 编译失败。

**spec 提示：** §18 Annotations 使用 `@` 表示装饰器/注解，但 §2.5 未将 `@` 列入 Operator 或 Punctuator。

**问题：** `@` 是一个语法标记（syntactic marker）但不是常规 Token？还是被归为某类 Token？

**对比：**
| 语言 | @ 字符 |
|------|-------|
| ArkTS | 仅用于注解（§18）|
| Java | 注解（@Override 等）|
| Swift | 属性包装器（@available 等）|
| TypeScript | 装饰器（@Component 等）|

**建议：** §2.5 明确 `@` 的归类（建议归入 Punctuator）。

---

### 问题 D：浮点字面量 f 后缀的优先级 ⭐ LOW

**实测：** `3.14f` 是合法 float 字面量（用例 016）。

**最长匹配疑问：** `3.14f` 是 `3.14`（double）+ `f`（标识符）还是单一 float 字面量？

**spec 行为：** 编译器正确识别为单一 float 字面量 Token。

**评价：** 实现质量良好，但 spec 应明确浮点后缀（如 `f`）的最长匹配优先级。

---

### 问题 E：连续 + 号的最长匹配解析 ⭐ LOW

**潜在场景：** `a+++b` 应解析为 `a++ + b` 还是 `a + ++b` 还是 `a + + + b`？

**实测：** 未直接测试此边界，但 spec 的最长匹配原则要求 tokenizer 优先匹配 `++`。

**建议：** 增加专项用例测试 `a+++b`、`a---b` 等边界。

## 四、改进方向建议

### 短期（spec 完善）
1. §2.5 增加 Operator/Punctuator 完整速查表
2. 明确 char 字面量前缀 `c` 在最长匹配中的优先级
3. 明确 `@` 字符的 Token 归类

### 中期（编译器/lint）
4. 提供 lint 规则：`a+++b` 等边界 Token 序列建议加空白
5. 改进错误信息：当数字后接字母时，提示是否需要空白分隔

### 长期（语言演进）
6. 评估是否需要支持自定义运算符（参考 Swift）
7. 持续跟踪 ECMAScript 新增 Token（如未来可能的 `?: ?? `）
