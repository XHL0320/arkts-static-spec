### 2.1 检视目标

1. **注释规范性检查**：检查所有 `.ets` 测试用例的注释是否符合模板（5 个 tag）
2. **issue_report.md 规范性**：检查是否符合 `03_Types/issue_report.md` 模板规范
3. **design_issues_report 规范性**：检查是否符合 `design_issues_report_3.2.md` 模板规范
4. **测试用例完整性**：检查是否覆盖 Spec 的所有规则
5. **跨语言对比完整性**：检查 `cross_lang_verify/` 是否完整
6. **ID 格式规范性**：检查用例 ID 和 issue ID 是否符合规范

## 四、详细检视步骤

### 4.1 每个子章节的检视步骤

```
步骤 1：检查目录结构
  ├─→ 是否存在 compile-pass/、compile-fail/、runtime/ 目录
  ├─→ 是否存在 cross_lang_verify/ 目录
  └─→ 是否存在设计报告文件（design_issues_report_15.X.md）

步骤 2：检查测试用例注释
  ├─→ 读取所有 .ets 文件
  ├─→ 检查是否包含 5 个 tag（@id, @expect, @section, @design, @note）
  ├─→ 检查 @id 是否符合命名规范（SEM_15_XX_YYY_CATEGORY_DESCRIPTION）
  └─→ 检查 @expect 是否正确（compile-pass / compile-fail / runtime）

步骤 3：检查 issue_report.md
  ├─→ 检查是否符合模板格式
  ├─→ 检查 ID 格式是否为 D-15.X-01 或 C-15.X-01
  ├─→ 检查表格是否包含 6 列（ID, Case, Symptom, Expected, Actual, Status）
  └─→ 检查异常详情是否包含问题描述、Spec 依据、跨语言对比

步骤 4：检查 design_issues_report_15.X.md
  ├─→ 检查是否符合模板格式（design_issues_report_3.2.md）
  ├─→ 检查是否包含报告头部、分类口径、行为差异、严重性总览、核心结论、建议改进
  └─→ 检查跨语言对比表格是否完整

步骤 5：检查跨语言对比
  ├─→ 检查 cross_lang_verify/ 目录是否存在
  ├─→ 检查是否有 Java 和 Swift 等价用例
  └─→ 检查是否有 verification_report.md

步骤 6：记录问题
  └─→ 将发现的问题记录到检视报告中
```

## 六、问题分类和处理

### 6.1 问题分类

| 分类 | 说明 | 处理方式 |
|------|------|----------|
| **格式问题** | 注释、ID、表格格式不符合规范 | 批量修复 |
| **内容问题** | 缺少必要内容（如跨语言对比） | 补充内容 |
| **设计问题** | 测试用例设计不合理 | 重新设计用例 |
| **Spec 不一致** | Spec 与实现不一致 | 记录到 issue_report 和 design_issues_report |

### 6.2 问题严重性

| 严重性 | 说明 | 响应时间 |
|--------|------|----------|
| **HIGH** | 影响测试用例执行或 Spec 一致性 | 立即修复 |
| **MEDIUM** | 影响文档规范性或完整性 | 近期修复 |
| **LOW** | 影响代码可读性 | 远期修复 |
