# 15.13 Static Initialization - ArkTS与Java/Swift/TS行为差异及规范一致性报告
---

**测试用例：** 10

## 一、行为差异与规范一致性概览

### 1. ArkTS 静态初始化与 Java 基本一致

ArkTS 静态初始化触发条件与 Java 类似：
- 静态字段访问触发
- 静态方法调用触发
- 类实例化触发

但 ArkTS 无显式 `static` 块（Java 有 `static { ... }`）。

### 2. 初始化顺序

ArkTS 静态字段按声明顺序初始化，与 Java 一致。

### 3. 递归初始化处理

ArkTS 处理递归静态初始化时，已初始化的字段不重复初始化。这与 Java 的递归初始化处理类似。

## 二、已验证的 ArkTS 规范一致行为

- 静态字段触发初始化 ✅
- 静态方法触发初始化 ✅
- `new` 触发静态初始化 ✅
- 命名空间触发初始化 ✅
- 模块变量触发初始化 ✅

## 三、设计观察

### 观察 1：ArkTS 无显式 static 块

Java 允许 `static { ... }` 块进行复杂静态初始化，ArkTS 不支持。ArkTS 的静态初始化仅限于字段初始化表达式。

### 观察 2：命名空间静态初始化

ArkTS 的命名空间（namespace）有独立的静态初始化语义，这是 TypeScript 兼容特性。

### 观察 3：SMART_GLOBAL 模式

测试用例显示 SMART_GLOBAL 模式下静态初始化行为可能不同，需要明确规范。

## 四、改进建议

### 建议 1：考虑支持显式 static 块

当前 ArkTS 静态初始化能力有限（仅字段初始化）。建议考虑支持显式 `static` 块，提高与 Java 的兼容性。

### 建议 2：明确命名空间静态初始化语义

命名空间静态初始化是 TypeScript 兼容特性，但规范说明不足。建议补充：
- 命名空间何时触发初始化
- 与模块初始化的关系
- 递归初始化的处理

### 建议 3：明确 SMART 模式与标准模式差异

SMART_GLOBAL/SMART_FUNC 模式行为可能与标准模式不同。建议明确差异并提供模式选择指南。
