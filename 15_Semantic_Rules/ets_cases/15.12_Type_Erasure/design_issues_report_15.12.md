## 一、行为差异与规范一致性概览

### 1. ArkTS 类型擦除与 Java 类似

ArkTS 在编译期擦除泛型类型参数，与 Java 类型擦除类似。但 ArkTS 的擦除规则更接近 TypeScript（保留部分类型信息用于编译期检查）。

### 2. FixedArray 擦除限制（ESE461884）⭐ ArkTS 特有

ArkTS 中 `FixedArray<T, N>` 在类型擦除后不保留长度信息 `N`，这与 Java 的数组类型擦除不同。这导致某些泛型场景下 `FixedArray` 无法使用。

### 3. 擦除后签名等效性判断

ArkTS 判断两个泛型函数是否等效时，会比较擦除后的签名。如果擦除后签名相同，则视为重复声明（编译错误）。这与 Java 类似，但判断规则更复杂（考虑 arity、数组嵌套层级等）。

### 4. 有效类型（Effective Type）计算

ArkTS 计算有效类型时：
- 字面量类型 → 基类型（`"foo"` → `string`）
- `union` → 共同超类型
- `array` → `array`
- `FixedArray` → `FixedArray`（但长度信息丢失）

## 二、已验证的 ArkTS 规范一致行为

- 泛型类型擦除基本规则 ✅
- 有效类型计算 ✅
- 擦除后签名等效性判断 ✅
- FixedArray 擦除限制 ✅
- 元组擦除 ✅
- 方法/静态方法/构造函数擦除 ✅

## 三、设计观察

### 观察 1：FixedArray 擦除限制影响 API 设计

ESE461884 导致 `Promise<FixedArray<T, N>>` 等 API 无法使用，因为擦除后 `FixedArray<T, N>` 的长度信息 `N` 丢失。建议：
- 明确 `FixedArray` 在擦除后的行为
- 考虑在规范中说明 `FixedArray` 擦除后的类型表示

### 观察 2：擦除后签名等效性判断规则复杂

当前 ARCHIVE 模式下有大量测试用例（021-031）验证擦除后签名等效性判断。规则涉及：
- 函数类型参数 arity
- 数组嵌套层级
- 元组 arity
- 方法/静态方法/构造函数

建议简化规则或在规范中明确说明。

### 观察 3：SMART 模式与 ARCHIVE 模式行为差异

测试用例显示 SMART 模式（SMART_GLOBAL, SMART_FUNC）与 ARCHIVE 模式行为不同。需要明确：
- 两种模式的差异
- 推荐使用场景
- 是否计划统一行为

## 四、改进建议

### 建议 1：明确 FixedArray 擦除语义

当前 `FixedArray<T, N>` 擦除后长度 `N` 丢失，导致某些泛型场景无法使用。建议：
- 在规范中明确 `FixedArray` 擦除后的类型表示
- 考虑是否保留长度信息（至少在某些上下文中）

### 建议 2：简化擦除后签名等效性判断规则

当前规则复杂，涉及多个维度（arity、数组嵌套、元组等）。建议：
- 简化等效性判断规则
- 在规范中明确说明判断逻辑
- 提供判定流程图或伪代码

### 建议 3：统一 SMART 模式与 ARCHIVE 模式

当前两种模式行为不一致，可能导致开发者困惑。建议：
- 明确两种模式的设计意图
- 考虑统一为一种模式
- 或提供清晰的模式选择指南

### 建议 4：增加类型擦除对运行时的影响说明

当前规范主要关注编译期擦除，对运行时影响说明不足。建议增加：
- 运行时类型信息是否可用
- 泛型反射支持
- 与 TypeScript 的 `design:type` 元数据对比

## 六、跨语言对比要点

| 维度 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 类型擦除时机 | 编译期 | 编译期 | 部分保留 | 无（reified） |
| 擦除后类型表示 | `any` | 边界类型或 `Object` | 保留 | 保留 |
| FixedArray 擦除 | 长度丢失 | N/A | N/A | N/A |
| 签名等效性判断 | 复杂 | 简单（arity + 类型） | 保留类型信息 | N/A |
| 运行时类型信息 | 不可用 | 部分可用（边界） | 可用 | 可用 |
