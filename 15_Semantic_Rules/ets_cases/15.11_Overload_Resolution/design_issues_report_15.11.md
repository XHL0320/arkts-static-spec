## 一、行为差异与规范一致性概览

### 1. 重载集不包含继承方法（与 Java 不同）

ArkTS 的重载集仅包含当前作用域声明的同名函数/方法，不包含继承的方法。这与 Java 不同（Java 中子类继承父类方法，构成重载集）。

### 2. W2323 警告（ArkTS 特有）

ArkTS 提供 W2323 警告，当重载集中 broad 类型参数隐藏 narrow 类型参数时发出。这是 ArkTS 特有的设计，帮助开发者发现潜在问题。

### 3. receiver 类型影响重载集

ArkTS 中方法调用的重载集动态构建，依赖于 receiver 的声明类型和 smart narrowing 类型。这与 Java（静态重载解析）和 Swift（重载解析规则不同）都有差异。

### 4. 返回类型不参与重载解析

ArkTS 与 Java 一致：仅参数类型参与重载解析，返回类型不参与。这与 C++ 不同（C++ 返回类型可参与重载解析）。

## 三、设计观察

### 观察 1：ArkTS 重载解析比 Java 更严格

Java 允许继承方法参与重载，ArkTS 不允许。这使得 ArkTS 的重载解析更可预测，但也限制了某些使用模式。

### 观察 2：W2323 警告的价值

W2323 警告（broad 隐藏 narrow）帮助开发者发现重载集设计问题。这与 TypeScript 的 `noUnreachableCode` 类似，但是编译警告而非错误。

### 观察 3：receiver 动态类型影响重载集

ArkTS 中 receiver 的 smart narrowing 类型可以影响重载集内容，这是 ArkTS 特有的设计（与 Java 静态解析不同）。

### 观察 4：ARCHIVE 模式行为差异

测试用例显示 ARCHIVE 模式下编译器行为更严格（如 SEM_15_11_041, SEM_15_11_042 拒绝 null/undefined 调用 object 方法）。需要确认这是设计意图还是实现限制。

## 四、改进建议

### 建议 1：明确重载集作用域规则

当前 spec 对"同一作用域"的定义不够明确。建议明确：
- 全局作用域 vs 模块作用域
- 类作用域 vs 实例作用域
- 继承链中哪些方法构成重载集

### 建议 2：统一 W2323 警告触发条件

当前 W2323 警告在某些场景下触发，某些场景下不触发。建议统一警告触发条件，并在 spec 中明确说明。

### 建议 3：增加 receiver 类型 narrowing 的规范说明

smart narrowing 如何影响重载集，当前 spec 说明不够详细。建议增加：
- 哪些 narrowing 技术影响重载集（instanceof, 显式类型转换等）
- narrowing 的作用域和生命周期
- 与声明类型的优先级关系

### 建议 4：明确 ARCHIVE 模式与 SMART 模式的差异

测试用例显示 ARCHIVE 模式与 SMART 模式行为不同。建议在 spec 中明确两种模式的差异和使用场景。
