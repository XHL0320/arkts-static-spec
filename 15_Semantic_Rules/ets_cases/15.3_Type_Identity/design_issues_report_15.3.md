## 二、与 Java/Swift/TypeScript 行为差异

### 2.1 类型系统差异
| 方面 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 类型系统 | 名义类型 | 名义类型 | 名义类型 | 结构类型 |
| 类型别名 | 支持，别名≡展开类型 | 不支持类型别名 | 支持 typealias≡原类型 | 支持，类型≡原类型 |
| 泛型同一性 | 相同参数≡同一类型 | 相同 | 相同 | 结构等价 |
| 联合类型 | 支持，顺序无关 | 不支持 | 不支持 | 支持，顺序无关 |
| 函数类型同一性 | 签名相同≡同一类型 | 方法签名 | 函数类型 | 结构等价 |

### 2.2 关键差异点
1. **ArkTS vs TypeScript**：
   - ArkTS 是名义类型系统，TypeScript 是结构类型系统
   - ArkTS 类型别名与其展开类型视为同一类型，TypeScript 也是（但基于结构等价）
   - ArkTS 泛型类型同一性基于类型参数同一性，TypeScript 基于结构等价

2. **ArkTS vs Java**：
   - Java 不支持类型别名，ArkTS 支持
   - Java 不支持联合类型，ArkTS 支持
   - 两者都是名义类型系统，泛型同一性判断一致

3. **ArkTS vs Swift**：
   - Swift 支持 typealias，与 ArkTS 的 type 类似
   - Swift 不支持联合类型，ArkTS 支持
   - 两者都是名义类型系统，类型同一性判断一致

## 三、规范措辞待澄清

### 3.1 联合类型同一性判断
- **问题**：规范未明确说明联合类型成员顺序是否影响同一性判断
- **建议**：明确说明联合类型成员顺序无关（set semantics）
- **当前实现**：ArkTS 编译器按 set semantics 处理，顺序无关

### 3.2 函数类型同一性判断
- **问题**：规范未明确说明函数参数名是否影响同一性判断
- **建议**：明确说明函数类型同一性仅基于参数类型和返回类型，参数名不影响
- **当前实现**：ArkTS 编译器仅检查参数类型和返回类型
