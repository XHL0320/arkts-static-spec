## 二、与 Java/Swift/TypeScript 行为差异

### 2.1 可赋值性语义对比
| 方面 | ArkTS | Java | Swift | TypeScript |
|------|-------|------|-------|------------|
| 子类型可赋值 | S<:T ⇒ S→T | 相同 | 相同 | 结构等价 |
| 数值拓宽 | int→double ✅ | ✅ | ❌ 需要显式转换 | ✅ |
| undefined 可赋值 | T\|undefined ← undefined | null→引用类型 | nil→Optional | T\|undefined←undefined |
| 接口可赋值 | 实现类→接口 ✅ | ✅ | ✅ | ✅ |
| 类型不匹配 | 编译错误 | 编译错误 | 编译错误 | 编译错误（严格模式） |

### 2.2 关键差异点
1. **ArkTS vs Swift**：
   - ArkTS 支持数值拓宽（int → double），Swift 需要显式转换
   - 这是 ArkTS 与 Java 保持一致的设计决策

2. **ArkTS vs TypeScript**：
   - ArkTS 是名义类型系统，可赋值性基于子类型关系
   - TypeScript 是结构类型系统，可赋值性基于结构等价
   - 但在基本的可赋值性规则上，两者一致

3. **ArkTS vs Java**：
   - 两者都是名义类型系统，可赋值性规则一致
   - ArkTS 支持联合类型和 undefined 可赋值性，Java 不支持

## 三、规范措辞待澄清

### 3.1 数值拓宽规则
- **问题**：规范未明确说明哪些数值类型转换是拓宽（widening），哪些是窄化（narrowing）
- **建议**：明确说明数值拓宽规则（如 int → double 是拓宽，double → int 是窄化）
- **当前实现**：ArkTS 编译器支持数值拓宽，窄化需要显式转换

### 3.2 undefined 可赋值性规则
- **问题**：规范未明确说明 undefined 可赋值性的完整规则
- **建议**：明确说明 undefined 可赋值给哪些类型（如 T | undefined、any 等）
- **当前实现**：ArkTS 编译器允许 undefined 赋值给含 undefined 的联合类型

### 3.3 赋值兼容性 vs 子类型关系
- **问题**：规范未明确说明可赋值性（Assignability）与子类型（Subtyping）的精确关系
- **建议**：明确说明可赋值性是子类型关系的子集，还是包含额外的规则（如数值拓宽）
- **当前实现**：ArkTS 可赋值性包含子类型关系和额外的拓宽规则
