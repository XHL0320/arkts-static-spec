## 设计问题清单

### 问题 1: 函数类型子类型规则的复杂性

**规范要求**:
- ArkTS 实施函数类型子类型规则：参数逆变，返回值协变
- 这是类型系统中最复杂的规则之一，增加了实现难度和用户理解难度

**对比分析**:

| 语言 | 函数类型子类型 | 参数规则 | 返回值规则 | 设计理由 |
|------|---------------|---------|-----------|----------|
| ArkTS | ✅ 支持 | 逆变 | 协变 | 类型安全 |
| Java | ✅ 支持 | 不变 | 不变 | 简单性优先 |
| Swift | ✅ 支持 | 逆变 | 协变 | 类型安全 |

**Java 示例** (函数类型不变):
```java
// Java: 函数类型不变（使用接口）
interface Function<F, T> {
    T apply(F f);
}
Function<Animal, Dog> f1 = ...;
Function<Dog, Animal> f2 = f1; // ❌ 编译错误：不变
```

**Swift 示例** (函数类型子类型):
```swift
// Swift: 函数类型子类型
let f1: (Animal) -> Dog = { animal in Dog() }
let f2: (Dog) -> Animal = f1 // ✅ 参数逆变，返回值协变
```

**可能的问题**:
1. 函数类型子类型规则复杂，用户可能难以理解
2. 参数逆变是反直觉的，需要详细解释

### 问题 2: 函数类型与面向对象的不兼容

**规范要求**:
- ArkTS 支持函数类型（从 TypeScript 继承）
- 但函数类型与面向对象编程（OOP）的设计哲学不完全兼容

**对比分析**:

| 语言 | 函数类型 | OOP 支持 | 兼容性 |
|------|----------|---------|--------|
| ArkTS | ✅ 支持 | ✅ 支持 | 中等（需要权衡） |
| Java | ✅ 支持（Lambda） | ✅ 支持 | 高（函数式接口） |
| Swift | ✅ 支持 | ✅ 支持 | 高（协议导向） |

**Java 示例** (函数式接口):
```java
// Java: 使用函数式接口
@FunctionalInterface
interface Function<F, T> {
    T apply(F f);
}
Function<Animal, Dog> f = (animal) -> new Dog();
```

**Swift 示例** (函数类型):
```swift
// Swift: 函数类型是一等公民
let f: (Animal) -> Dog = { animal in Dog() }
```

**可能的问题**:
1. 函数类型与 OOP 的设计哲学不完全兼容
2. 可能导致代码风格不一致（函数式 vs 面向对象）

**改进建议**:
1. 在规范中说明函数类型的使用场景和限制
2. 提供最佳实践，帮助用户选择合适的编程范式
3. 考虑增加模式匹配（pattern matching）支持，提高函数类型的可用性

**优先级**: 低
