## 3. 工作流程（实际执行版）

> **重要原则**：禁止未确认的批量操作，每步处理前需确认。修订完成后必须验证。

### 步骤 1：审计现有 md 文件

**操作**：
1. 使用 `list_dir` 列出目标目录下的所有文件
2. 使用 `read_file` 读取所有现有 md 文件
3. 识别共性问题：
   - 标题重复/格式错误
   - 未替换的占位符变量（`$p`, `$f`, `$r`, `$t`, `$pass`, `gap_count`, `YYYY-MM-DD` 等）
   - 测试用例列表与实际测试文件不匹配
   - 内容不完整（缺少某些测试用例的说明）

**输出**：审计报告，列出所有发现的问题

**确认点**：向用户展示审计报告，确认问题清单

---

### 步骤 2：按需创建缺失文件

**操作**：
1. 检查是否缺少 4 个必填 md 文件
2. 如果缺少，参考模板创建：
   - `test_design_mindmap_XX.md`
   - `test_report_XX.md`
   - `design_issues_report_XX.md`
   - `comparison_report_arkts_java_swift.md`

**注意**：
- 文件名格式必须正确（`XX` 替换为实际的小节编号，如 `15.11.1`）
- 文件内容必须遵循模板格式（参考 `doc/03_Types/ets_cases/3.11_Type_void_or_undefined/`）

**确认点**：向用户展示创建的文件列表，确认无误

---

**操作**：
1. 使用 `list_dir` 列出 `compile-pass`、`compile-fail`、`runtime` 目录下的所有 `.ets` 文件
2. 使用 `read_file` 读取所有测试用例文件
3. 统计测试用例数量和类型：

**目的**：
- 了解测试用例的具体内容

### 4.1 文件命名规范

**md 文件命名**：
- `test_design_mindmap_XX.md`（`XX` 替换为小节编号，如 `15.1.2`）
- `test_report_XX.md`
- `design_issues_report_XX.md`

**用例文件命名**：
- 格式：`SEM_XX_XX_XXX_PASS/FAIL/RUNTIME_描述.ets`
- 示例：`SEM_15_01_006_PASS_KNOWN_RHS_ASSIGN.ets`
- 编号规则：`SEM` + `章节号` + `3位序号` + `类型` + `描述`

**注意**：
- 如果用例文件名格式不一致（如 `SEM_15_01_06_001.ets`），保持原文件名不变，只在 md 文件中正确引用。

---

### 4.2 占位符替换

**常见占位符**：
- `$p` → 实际用例数量（compile-pass）
- `$f` → 实际用例数量（compile-fail）
- `$r` → 实际用例数量（runtime）
- `$t` → 总用例数量
- `$pass` → 通过率
- `gap_count` → 缺失用例数量
- `YYYY-MM-DD` → 实际日期
- `XX.X` → 实际小节编号

**替换规则**：
- 所有占位符必须替换为实际值
- 不能保留任何占位符

## 6. 常见问题与解决方案

### Q1: 用例文件名格式不一致怎么办？

**A**: 保持原文件名不变，只在 md 文件中正确引用。不要重命名用例文件，除非用户明确要求。

---

### Q2: 测试用例缺失怎么办？

**A**: 在 `design_issues_report_XX.md` 中记录缺失的测试用例，并在 `test_design_mindmap_XX.md` 的"覆盖情况"中标记为"待测试"。

---

### Q3: 跨语言对比报告缺少 Java 或 Swift 代码怎么办？

**A**: 根据 ArkTS 用例，手动编写对应的 Java 和 Swift 代码。确保代码功能等价，便于对比。

---

### Q4: 实测结果无法验证怎么办？

**A**: 在报告中明确说明"未实测"，并提供理论分析结果。标注"待实测验证"。

---

### Q5: md 文件内容过长怎么办？

**A**: 保持内容完整，不删减。如果确实过长，可以考虑拆分为多个文件（但不建议）。

### 9.2 关键发现

#### 9.2.1 15.2 Subtyping（子类型关系）

**15.2 Subtyping（总述）**：
1. **自身子类型正确**：ArkTS 正确支持 `T <: T`（任何类型都是自身的子类型）
2. **子类型拒绝正确**：不兼容的类型之间不能赋值
3. **子类型传递性正确**：如果 `A <: B` 且 `B <: C`，则 `A <: C`

**15.2.1 Subtyping for Non-Generic Classes and Interfaces**：
1. **类继承子类型正确**：子类是父类的子类型
2. **接口实现子类型正确**：实现类是接口的子类型
3. **与 Java 一致**：类继承子类型行为与 Java 一致

**15.2.2 Subtyping for Generic Classes and Interfaces**：
1. **泛型不变性正确**：`Holder<Derived>` 不是 `Holder<Base>` 的子类型
2. **与 Java 一致**：泛型不变性行为与 Java 一致
3. **与 TypeScript 差异**：TypeScript 支持泛型协变/逆变（使用 `out`/`in`）

**15.2.3 Subtyping for Literal Types**：
1. **字面量类型子类型正确**：字面量类型是其基础类型的子类型
2. **与 TypeScript 一致**：字面量类型子类型行为与 TypeScript 一致

**15.2.4 Subtyping for Tuple Types**：
1. **元组类型子类型正确**：元组类型协变（元素类型兼容时）
2. **与 TypeScript 一致**：元组类型子类型行为与 TypeScript 一致

**15.2.5 Subtyping for Union Types**：
1. **联合类型子类型正确**：`A <: A | B`（A 是 A | B 的子类型）
2. **与 TypeScript 一致**：联合类型子类型行为与 TypeScript 一致

**15.2.6 Subtyping for Function Types**：
1. **函数类型子类型正确**：参数逆变，返回值协变
2. **与 TypeScript 一致**：函数类型子类型行为与 TypeScript 一致

**15.2.7 Subtyping for Fixed Size Array Types**：
1. **固定大小数组类型不变**：`FixedArray<Derived>` 不是 `FixedArray<Base>` 的子类型
2. **与 Java 一致**：数组类型行为在 Java 中是协变，ArkTS 中是不变

**15.2.8 Subtyping for Intersection Types**：
1. **交叉类型未实现**：编译器未实现交叉类型（已知 GAP，跟踪号 ESY145527）

**15.2.9 Subtyping for Difference Types**：
1. **差分类型未实现**：编译器未实现差分类型（已知 GAP，跟踪号 ESY145527）

#### 9.2.2 15.3 Type Identity（类型同一性）

3. **验证必须执行**：修订完成后必须执行验证步骤
4. **记录所有操作**：记录所有操作和决策，便于回溯

**效果**：采用严格的分步验证工作流程后，避免了问题复发。

#### 10.1.2 占位符替换的常见错误

**问题**：许多 md 文件中存在未替换的占位符变量。

**常见占位符**：
- `$p` → 实际用例数量（compile-pass）
- `$f` → 实际用例数量（compile-fail）
- `$r` → 实际用例数量（runtime）
- `$t` → 总用例数量
- `$pass` → 通过率
- `gap_count` → 缺失用例数量
- `YYYY-MM-DD` → 实际日期
- `XX.X` → 实际小节编号

**解决方案**：
1. **审计时重点检查**：在步骤 1（审计）时，使用 `search_content` 工具搜索所有占位符
2. **修订时逐个替换**：在步骤 4（修订）时，逐个替换所有占位符
3. **验证时再次检查**：在步骤 5（验证）时，再次检查是否有残留占位符

**检查命令**：
```bash
# 搜索所有占位符
search_content <md_file> pattern="\$p|\$f|\$r|\$t|\$pass|gap_count|YYYY-MM-DD"
```

**问题**：md 文件中的测试用例列表与实际测试文件不匹配，导致内容不完整。

3. **验证时检查一致性**：验证时检查 md 文件中的测试用例清单是否与实际文件一致

### 10.2 最佳实践

#### 10.2.1 工作流程最佳实践

1. **先审计，后修订**：不要直接修订，先审计现有文件，识别所有问题
2. **逐文件确认**：每修订完一个文件，向用户展示修订内容，确认无误
3. **验证不可跳过**：修订完成后必须执行验证步骤，不能跳过
4. **记录所有操作**：记录所有操作和决策，便于回溯

#### 10.2.2 内容编写最佳实践

1. **标题格式统一**：严格遵循 `# XX.X 标题 - 报告类型` 格式

3. **代码块格式正确**：代码块必须指定语言（如 `typescript`、`java`、`swift`）

### 10.3 常见问题与解决方案（更新版）

#### Q1: 用例文件名格式不一致怎么办？

**A**: 保持原文件名不变，只在 md 文件中正确引用。不要重命名用例文件，除非用户明确要求。

---

#### Q2: 测试用例缺失怎么办？

**A**: 在 `design_issues_report_XX.md` 中记录缺失的测试用例，并在 `test_design_mindmap_XX.md` 的"覆盖情况"中标记为"待测试"。

---

#### Q3: 跨语言对比报告缺少 Java 或 Swift 代码怎么办？

**A**: 根据 ArkTS 用例，手动编写对应的 Java 和 Swift 代码。确保代码功能等价，便于对比。

---

#### Q4: 实测结果无法验证怎么办？

**A**: 在报告中明确说明"未实测"，并提供理论分析结果。标注"待实测验证"。

---

#### Q5: md 文件内容过长怎么办？

**A**: 保持内容完整，不删减。如果确实过长，可以考虑拆分为多个文件（但不建议）。

---

#### Q6: 修订后出现格式错误怎么办？

**A**: 使用 `read_lints` 工具检查文件，修复所有格式错误。常见格式错误包括：
- 表格列对齐错误
- 代码块缺少闭合标记
- 列表缩进错误

---

#### Q7: 如何避免批量操作错误？

**A**: 严格遵循以下原则：
1. 禁止未确认的批量操作
2. 每步处理前需确认
3. 逐文件修订，逐文件确认
4. 修订完成后必须验证
