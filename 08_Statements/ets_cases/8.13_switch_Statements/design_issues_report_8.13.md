# 8.13 switch 语句 - ArkTS 设计问题发现报告

**报告日期：** 2026-06-18
**测试用例数：** 11
**目的：** 通过用例执行（编译期 + 运行时）+ 跨语言对比，识别 ArkTS 静态语言的设计问题。

---

## 一、新发现的设计问题（基于规范分析）

### 问题 1：规范 TODO -- case 中使用非字面量常量表达式会导致断言错误 ⭐ MEDIUM
**用例：** 无（尚未编写测试用例）
**实际行为/预期行为：** ArkTS 规范本身标记了一个 TODO 条目：当 case 标签使用非字面量表达式（如变量、函数调用或计算表达式）时，编译器可能触发断言错误（assertion error），而非给出规范的编译期错误诊断。规范未明确定义合法的 case 表达式范围——是仅允许字面量常量，还是遵循类似 Java 常量表达式的规则允许编译期常量表达式。

**对比：**

| 语言 | 行为 |
|------|------|
| **Java (JLS SE21)** | 明确定义可在 case 标签中使用的"常量表达式"，包括字面量、编译期常量（以常量表达式初始化的 static final 变量）以及枚举常量。非常量表达式会被拒绝并给出清晰的编译期错误信息。 |
| **Swift (5.x)** | Swift 中的 case 模式不限于常量——可以包含范围、元组、类型注解以及 `where` 子句。这是一种本质上不同的（更富表达力的）模式匹配系统。 |
| **ArkTS** | 规范对此问题标记了 TODO，行为未定义；非字面量 case 表达式可能触发断言错误而非编译期错误。 |

**评价/建议：** 规范应明确定义哪些表达式可作为 case 标签使用。合理的方案是遵循 Java 的常量表达式规则：字面量以及具名编译期常量（以字面量或常量表达式初始化的 final/const 变量）。非常量 case 表达式应产生清晰的编译期错误，而非断言错误。

### 问题 2：规范 TODO -- 仅含 default 子句的 switchBlock 行为缺陷 ⭐ HIGH
**用例：** 无（尚未编写测试用例）
**实际行为/预期行为：** ArkTS 规范标记了另一个 TODO：当 switch 块仅包含一个 `default` 子句（无 `case` 子句）时，default 中的语句/代码块**不会被实际执行**。这违反了规范自身的语义规则：如果未发生匹配且存在 default 子句，执行应转移到 default 子句的语句。在仅含 default 的 switch 中，无须匹配任何 case 子句，因此 default 应始终执行。

**对比：**

| 语言 | 行为 |
|------|------|
| **Java (JLS SE21)** | 仅含 `default` 的 switch 正确执行 default 子句。 |
| **Swift (5.x)** | 仅含 `default:` 的 switch 合法且执行其代码体。（注：Swift 仍要求 switch 穷尽；default 子句满足此要求。） |
| **ArkTS** | 已知运行时缺陷：仅含 default 的 switch 块不执行 default 语句。 |

**评价/建议：** 这是一个严重缺陷，应在 ArkTS 运行时中修复。仅含 default 的 switch 中的 default 子句必须执行其语句。应添加对应的测试用例，并在运行时修复后进行验证。

---

### 问题 3：不要求穷尽性（与 Swift 不同） ⭐ LOW
**用例：** 所有 compile-pass 用例
**实际行为/预期行为：** ArkTS 不要求 switch 语句穷尽（即覆盖 switch 表达式的所有可能值）。若无 case 匹配且无 default 子句，执行静默地无操作穿透。这与 Java 的传统 switch 语义一致，但与 Swift 显著不同。

**对比：**

| 语言 | 行为 |
|------|------|
| **Java (JLS SE21)** | 传统 switch 不要求穷尽性。增强 switch（`->` 语法）和 switch 表达式要求穷尽性。 |
| **Swift (5.x)** | 编译期要求穷尽性。所有可能值必须被 case 子句覆盖，或必须存在 default 子句。这防止了一类运行时 bug——未处理的值被静默忽略。 |
| **ArkTS** | 不检查穷尽性。无匹配且无 default 时静默穿透，不产生任何效果。 |

**评价/建议：** 建议考虑为有限值集合类型（尤其是 boolean 和 enum 类型）的非穷尽 switch 语句添加可选的编译器警告或 lint 规则。对于没有 default 子句的 boolean switch，编译器可以警告仅覆盖了 `true` 和 `false` 之一（当前 STM_08_13_004 同时覆盖了 `true` 和 `false`）。

---

### 问题 4：显式允许重复 case 标签 ⭐ LOW
**用例：** 无（尚未编写测试用例）
**实际行为/预期行为：** ArkTS 规范明确指出允许重复的 case 标签：`case null:` / `case null: // 可以有多个具有相同表达式的 case 子句`。这允许存在多个具有相同表达式值的 case 子句。虽然语法上合法，但这是不寻常的设计：由于始终进入第一个匹配的 case，后续重复 case 将永远不可达。

**对比：**

| 语言 | 行为 |
|------|------|
| **Java (JLS SE21)** | 重复的 case 标签产生编译期错误。 |
| **Swift (5.x)** | 重复的模式产生编译期错误（警告该模式永远不会被匹配）。 |
| **ArkTS** | 显式允许重复的 case 标签，无错误或警告。 |

**评价/建议：** 重复的 case 标签应产生编译期错误或至少产生警告，因为这表示死代码且极可能是程序员的编程错误。

---

### 问题 5：默认穿透（fall-through）语义，无 `fallthrough` 关键字 ⭐ LOW
**用例：** STM_08_13_002, STM_08_13_010
**实际行为/预期行为：** ArkTS 采用 Java 的传统穿透语义：若 case 子句不以 `break` 结束，执行将穿透至下一个 case 子句。这是传统的 C 风格行为，广为人知但容易出错。

**对比：**

| 语言 | 行为 |
|------|------|
| **Java (JLS SE21)** | 传统 switch 具有穿透行为。Java 14+ 增加了 `->` switch 标签，使用单个表达式/代码块且无穿透。使用 `break` 关键字退出。 |
| **Swift (5.x)** | 无隐式穿透。case 执行完毕后自动退出 switch。需要显式 `fallthrough` 关键字才能继续到下一个 case。这被广泛认为是更好的设计，因为它防止了意外的穿透 bug。 |
| **ArkTS** | 采用传统 C/Java 风格隐式穿透；无 `fallthrough` 关键字。 |

**评价/建议：** 虽然改变此行为会破坏与 TypeScript/JavaScript 传统的兼容性，但添加编译器警告——当检测到穿透（下一个 `case` 标签之前无 `break`）时——将有助于防止常见 bug。

## 四、跨语言对比结论

ArkTS 的 switch 语句在整体设计上与 **Java 传统 switch** 最为接近，而非 Swift 的模式匹配系统。关键分歧点总结如下：

| 特性 | ArkTS | Java (传统 switch) | Swift (5.x) |
|------|-------|-------------------|------------|
| case 穿透行为 | 隐式穿透（需 break 终止） | 隐式穿透（需 break 终止） | 无隐式穿透（需显式 fallthrough） |
| 穷尽性检查 | 无 | 无（增强 switch 有） | 编译期强制穷尽 |
| 重复 case 标签 | 允许 | 编译期错误 | 编译期错误 |
| 仅含 default 的 switch | 已知运行时缺陷 | 正确执行 | 正确执行 |
| case 表达式类型 | 未明确定义（有 TODO） | 明确定义（常量表达式） | 模式匹配（范围、元组等） |
| 类型不匹配 case | 编译期错误 | 编译期错误 | 编译期错误 |
| 带标签的 break | 支持 | 支持 | 不支持（语言无 goto 式跳转） |

**整体评价：** ArkTS switch 语句继承了 TypeScript/JavaScript 的 C 风格语义基础，并在类型安全方面对齐 Java 的方向。但存在一个 ⭐ HIGH 严重性的已知运行时缺陷（仅含 default 的 switch 不执行）和一个 ⭐ MEDIUM 的规范未定义行为（非字面量 case 表达式）。其余设计选择（穿透语义、不强制穷尽性、允许重复标签）属于保守的兼容性决策——保持了与 C/Java/TypeScript 传统的熟悉感，但未采纳 Swift 等现代语言中已被验证更安全的 switch 设计模式。

## 五、改进方向建议

### 短期（需立即修复）

1. **修复仅含 default 的 switch 运行时缺陷**：当 switch 块仅包含 default 子句时，default 语句必须被执行。这是直接的规范违规，应作为最高优先级缺陷处理。
2. **定义合法的 case 表达式**：明确规范中 case 标签可接受的表达式形式（建议采用 Java 风格的常量表达式规则），并将非字面量 case 表达式的断言错误改为规范的编译期错误诊断。

### 中期（建议增强）

3. **添加编译器警告：检测穿透**：当 case 子句不以 `break` 结束时（即存在隐式穿透），发出编译器警告。这可以在不改变语言语义的前提下帮助捕获常见编程错误。
4. **添加编译器警告：重复 case 标签**：当同一 switch 中出现重复的 case 表达式值时，发出编译期警告或错误，以标识死代码。
5. **添加 lint/警告：非穷尽 switch**：对 boolean 和 enum 类型的 switch 表达式，当未覆盖所有可能值且无 default 子句时，发出警告。

### 长期（设计层面考虑）

6. **评估引入无穿透 switch 形式**：参考 Java 14+ 的 `->` 语法或 Swift 的自动退出行为，考虑增加一种不依赖 `break` 的 case 形式，以减少穿透相关的编程错误。
7. **补齐测试覆盖盲区**：添加以下场景的测试用例：
   - 仅含 default 子句的 switch（无 case 子句）
   - 含非字面量 case 表达式的 switch
   - null/undefined switch 表达式与 null case 标签
   - 空 switch 块（无 case 且无 default）
   - 重复 case 标签（验证当前允许行为）
   - 类/对象类型 switch 表达式
   - 3 层以上嵌套 switch 中的 break/continue 交互
   - int 边界值（MIN_VALUE, MAX_VALUE）作为 switch 表达式
