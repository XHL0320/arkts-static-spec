## 四、跨语言对比结论

### 对比 A：finally 中抛出异常覆盖原始异常（规则4）

ArkTS 规则4 规定当 finally 子句异常完成时，整个 try 语句异常完成，且 finally 中的异常会覆盖 catch 已处理的原始异常。

| 语言 | 行为 | 一致性 |
|------|------|--------|
| **ArkTS** | finally 异常完成时，try 语句异常完成，原始异常被覆盖 | - |
| **Java** | 完全相同。JLS SE21 §14.20.2 规定若 finally 块突然完成，则 try 语句因该原因突然完成，原因被替换 | ✅ 一致 |
| **Swift** | 无 finally 关键字，使用 `defer` 语句。defer 块中若抛出错误且同时有另一错误在传播中，会导致运行时错误 | ❌ 架构不同 |

**评价：** ArkTS 与 Java 在该规则上完全一致。Swift 的 defer 设计从根本上不同于 finally，无法直接对比。

### 对比 B：错误传播机制（规则3）

ArkTS 规则3 规定无 catch 子句时异常传播到外围作用域或调用者作用域，直到找到匹配的 catch。

| 语言 | 行为 | 一致性 |
|------|------|--------|
| **ArkTS** | 异常沿作用域链向外传播，无需函数签名声明 | - |
| **Java** | 完全相同的错误传播机制（不要求函数签名声明 throws） | ✅ 一致 |
| **Swift** | 必须显式标记 `throws` 函数签名，错误只能沿调用链通过 `throws` 传播，所有未处理错误在编译期标记 | ❌ 架构不同 |

**评价：** ArkTS 与 Java 在错误传播机制上完全一致。这既是简化也是风险——调用者可能不知道某个函数会抛出错误。Swift 的显式 throws 设计更安全但更冗余。

### 对比 C：缺少 try-with-resources 机制

Java JLS SE21 §14.20.3 定义了 try-with-resources 语句，用于自动关闭实现了 `AutoCloseable` 的资源。ArkTS 没有对应的机制。

**评价：** ArkTS 作为静态类型脚本语言，其运行时环境（OpenHarmony）的资源管理模型可能不依赖此机制。这是合理的语言设计取舍，不属于设计缺陷。

### 对比 D：try 语句强制要求 catch 或 finally

ArkTS 要求 try 语句必须至少包含 catch 或 finally 子句（用例 005 验证）。这与 Java 完全一致。单独 `try { }` 后无 catch 也无 finally 是语法错误。

**评价：** 与 Java 一致，合理的设计决策。

## 五、改进方向建议

### 短期建议

- **无。** 当前 try 语句的执行语义与规范完全一致，无需要立即修复的设计问题。

### 中期建议

- **考虑引入显式错误传播标记（类似 Swift throws）：** 当前 ArkTS 与 Java 一样，不需要在函数签名中声明可能抛出的异常。这在大型项目中可能导致调用者无法静态感知潜在的错误源。可考虑在中期引入可选的 throws 注解或类型系统扩展，提供编译期错误路径检查能力。

### 长期建议

- **评估 try-with-resources 的适用性：** 若 OpenHarmony 生态中资源管理（如文件句柄、网络连接）的自动清理需求增加，可评估引入类似 Java try-with-resources 的语法糖，提升资源管理的安全性和简洁性。
