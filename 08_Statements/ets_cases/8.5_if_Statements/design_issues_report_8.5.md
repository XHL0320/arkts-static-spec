### 规范评估

ArkTS 对 if 语句的定义（第 8.5 节）是一种清晰、类型安全的条件构造：

- **语法**：`ifStatement: 'if' '(' expression ')' thenStatement ('else' elseStatement)?` —— 遵循传统的 C 家族语法，无任何偏离。
- **Boolean 条件强制检查**：规范要求条件表达式必须为 `boolean` 类型（或扩展条件表达式类型）。这消除了 JavaScript 中常见的将非布尔值（`0`、`""`、`null`、`undefined`、对象）隐式转换为布尔值的陷阱。ArkTS 在编译期进行的严格类型检查可防止此类错误。
- **悬空 else 解析**：规范将每个 `else` 绑定到最近的未匹配 `if`，这与 C、C++、Java、C# 和 Swift 中使用的标准解析规则一致。此处不存在歧义或设计问题。
- **块作用域语义**：使用大括号（`{}`）时，会创建块作用域；不使用大括号时，不会创建块作用域。这与第 8.3 节中定义的块作用域规则一致，并符合其他静态类型语言的预期。

全部 3 个编译失败用例（STM_08_05_006 至 STM_08_05_008）正确验证了将非布尔类型（`int`、`string`、`number`）用作 if 条件会在编译期产生错误，这正是规范所要求的。

## 四、跨语言对比结论

| 语言 | 非布尔条件行为 | 类型安全检查 |
|------|--------------|-------------|
| ArkTS | ❌ 编译期错误 | 严格，仅允许 `boolean` |
| TypeScript/JavaScript | ✅ 隐式转换（truthy/falsy） | 宽松 |
| C/C++/Java/C# | ✅ 允许（整数/对象判空） | 部分检查 |

ArkTS 采用了比 TypeScript 和 JavaScript 更严格的布尔条件检查策略，在编译期即捕获隐式类型转换错误，提高了代码的健壮性和可读性。这一设计与 Swift 的布尔条件强制要求最为接近。
