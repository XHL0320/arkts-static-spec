## 四、跨语言对比结论

ArkTS 8.11 规范中定义的 `continue` 语句遵循与 TypeScript/JavaScript、Java（JLS SE21）和 Swift 5.x 中相同的语义，未引入任何超出规范定义的 ArkTS 特有偏差或限制。

| 方面 | ArkTS 8.11 | Java (JLS SE21) | Swift 5.x | 问题？ |
|------|------------|------------------|-----------|--------|
| 无标签 continue | 跳转到最内层外层循环的下一次迭代 | 跳转到最内层外层循环的下一次迭代 (14.16) | 跳转到最内层循环的下一次迭代 | 无 |
| 带标签 continue | `continue identifier` 跳转到被标签标记的循环的下一次迭代 | `continue identifier` 跳转到被标签标记的循环的下一次迭代 (14.16) | `continue labelName` 相同语义 | 无 |
| 编译期错误：循环外使用 | 要求 | 要求 (14.16："必须包含在循环语句中") | 要求 | 无 |
| 编译期错误：标签不匹配 | 要求（标签必须匹配外层循环） | 要求 (14.16："标签必须标记外层循环语句") | 要求 | 无 |

### 具体设计要点

1. **一致的标签语义**：ArkTS 要求 `continue` 标签必须指向一个循环语句，这与 Java 的要求（JLS SE21 14.16：标签"必须标记一个外层循环语句"）和 Swift 的行为（标签只能放在循环语句之前）完全一致。无偏差。

2. **无新增限制**：ArkTS 没有在规范从标准实践中得出的约束之外添加任何额外的编译期约束。三种循环类型（for、while、do-while）均接受 `continue`，与 Java 和 Swift 保持一致。

3. **无安全性放松**：与某些语言可能在 `continue` 位于循环外时仅发出警告而不会报错的做法不同，ArkTS 正确地强制执行编译期错误，与 Java 的严格处理方式一致。

4. **无隐藏行为**：`continue` 语句不涉及隐式类型强制转换，无副作用，且除了标准的控制流外，不与 ArkTS 的其他特性（如 nullish 类型、async）产生特殊交互。

## 五、改进方向建议

规范 8.11 中 `continue` 语句的定义清晰、完备，当前无需改进。该规范是 TypeScript、Java 和 Swift 通用行为的干净、明确子集，不存在设计问题、不一致性或歧义。
