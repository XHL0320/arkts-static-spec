# 7.4.1 Function Reference — ArkTS 与 Java/Swift/TS 行为差异及规范一致性报告

## 设计问题及差异清单

### ID-01: 显式重载语法（ArkTS 独有）⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_01_005_PASS_INDIVIDUAL_OVERLOAD_REF / EXP_07_04_01_008_FAIL_EXPLICIT_OVERLOAD_NAME |
| **实测结果** | `overload foo { foo1, foo2 }` 中 `foo` 名称作为引用报编译时错误，`foo2` 作为引用通过 |
| **错误信息** | 编译时错误（使用 `foo` 时） |

**描述**：ArkTS 支持 `overload foo { foo1, foo2 }` 显式重载声明语法，Java 和 Swift 均无等价概念。Java 中所有重载都是隐式的（同名不同参方法），Swift 同理。ArkTS 显式重载名称 `foo` 不能作为函数引用（编译时错误），但单个组件函数 `foo2` 可以。Java/Swift 中重载函数的歧义通过目标类型消除，而非禁止引用。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `let y = foo` | ❌ 编译时错误（重载名称） |
| ArkTS | `let y = foo2` | ✅ 引用单个函数 |
| Java | N/A | ❌ 无此语法 |
| Swift | N/A | ❌ 无此语法 |

**分类**：语法级别的语言设计差异（ArkTS 独有特性），符合 ArkTS 规范的设计差异

**建议**：当前设计无缺陷，显式重载是 ArkTS 为代码可读性增加的显式声明机制。

### ID-02: 泛型函数引用约束（ArkTS vs Java）⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_01_004_PASS_GENERIC_FUNCTION_REF / EXP_07_04_01_006_FAIL_GENERIC_REF_NO_ARGS |
| **实测结果** | 泛型函数引用 `gen<string>` 编译通过，`gen` 无类型参数报编译时错误 |
| **错误信息** | 编译时错误（无类型参数时） |

**描述**：ArkTS 要求泛型函数引用必须携带显式类型参数，否则编译时错误。Java 中泛型方法引用依赖目标类型推断，不要求显式类型参数。Swift 与 ArkTS 一致，需要类型标注。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `let b = gen` | ❌ 编译时错误 |
| Java | `Function<String, String> f = Class::<String>identity` 或简写 | ✅ 目标类型推断 |
| Swift | `let f = identity`（无上下文） | ❌ 编译时错误 |

**分类**：编译时安全检查差异，符合 ArkTS 规范的设计差异

**建议**：当前设计合理，与 Swift 主流实践一致。

### ID-03: 顶层函数引用（ArkTS vs Java）⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_01_001_PASS_BASIC_FUNCTION_REF |
| **实测结果** | `let func = foo` 编译通过 |
| **错误信息** | 无 |

**描述**：ArkTS 和 Swift 支持顶层函数引用，Java 所有方法必须属于类，只能用静态方法引用 `Class::method` 模拟。

**跨语言对比**：

| 语言 | 语法 | 灵活性 |
|------|------|--------|
| ArkTS | `let func = foo` | 直接 |
| Java | `Function<T,R> f = Class::method` | 间接（需函数式接口） |
| Swift | `let func = foo` | 直接 |

**分类**：语言结构差异，符合各自语言设计

**建议**：无需调整。

### ID-04: 重载函数名引用歧义消除策略 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_01_007_FAIL_OVERLOADED_FUNC_NAME |
| **实测结果** | 隐式重载函数名作为引用报编译时错误 |
| **错误信息** | 编译时错误 |

**描述**：三语言都不允许重载函数名直接作为引用而不消除歧义，但消除方式不同。ArkTS 最严格，直接禁止引用重载函数名。

**跨语言对比**：

| 语言 | 消除策略 |
|------|---------|
| ArkTS | 直接禁止引用重载函数名（最严格） |
| Java | 通过目标函数式接口类型自动消除歧义 |
| Swift | 通过显式类型标注消除歧义 |

**分类**：编译时重载解析策略差异，符合各自语言设计

**建议**：ArkTS 策略最安全，不推荐放宽。
