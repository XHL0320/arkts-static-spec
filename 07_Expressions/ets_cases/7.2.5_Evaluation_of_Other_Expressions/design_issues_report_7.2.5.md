# 7.2.5 Evaluation of Other Expressions — ArkTS 与 Java/Swift 行为差异及规范一致性报告

## 设计问题及差异清单

### ID-01: Swift 数组为值类型 vs ArkTS/Java 引用类型 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_02_05_011_RUNTIME_INDEX_ASSIGN_ORDER |
| **实测结果** | ArkTS 和 Java 数组为引用类型，Swift 数组为值类型（COW） |
| **差异类型** | 符合各自语言设计哲学的架构差异 |

**描述**：ArkTS 和 Java 的数组是引用类型：变量持有对象引用，函数返回数组时返回引用，下标赋值修改原数组。Swift 的数组是值类型（COW），函数返回数组时返回不可变副本，无法对返回值直接做下标赋值。为在 Swift 中验证相同语义，需使用 `class` 包装数组。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `getArr("A")[getIdx("I")] = getVal("V")` | ✅ 修改原数组 |
| Java | `getSArr("A")[getIdx("I")] = getVal("V")` | ✅ 修改原数组 |
| Swift | `getSArrRef("A")[getIdx("I")] = getVal("V")` | ✅ 使用 class 包装实现引用语义 |

**分类**：符合各自语言设计哲学的架构差异（非 Spec 不一致）

**建议**：无需修复。在跨语言验证时，Swift 用例需使用 `class` 包装数组以模拟引用语义。

### ID-02: Lambda/Closure 惰性求值三语言一致 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_02_05_012_RUNTIME_LAMBDA_LAZY |
| **实测结果** | 三语言行为完全一致 |
| **差异类型** | 三语言行为一致 |

**描述**：ArkTS、Java、Swift 在 Lambda/Closure 的惰性求值上行为完全一致：创建时体不被求值，仅在调用时执行。重复调用时体每次都被重新求值。

**跨语言对比**：

| 语言 | 创建时体执行？ | 调用时体执行？ | 重复调用体重新执行？ |
|------|--------------|--------------|-------------------|
| ArkTS | ❌ 不执行 | ✅ 执行 | ✅ 每次重新执行 |
| Java | ❌ 不执行 | ✅ 执行 | ✅ 每次重新执行 |
| Swift | ❌ 不执行 | ✅ 执行 | ✅ 每次重新执行 |

**分类**：三语言行为一致（无差异）

**建议**：无。
