# 7.1.1 Operator Precedence — ArkTS 与 Java/Swift/TS 行为差异及规范一致性报告

## 设计问题及差异清单

### ID-01: 赋值给非左值编译通过 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_01_01_021_FAIL_ASSIGN_TO_NON_LVALUE |
| **实测结果** | 编译通过（期望编译时错误） |
| **错误信息** | 无错误 |

**描述**：根据 ArkTS spec 运算符优先级，`+` 优先级高于 `=`，`a + b = c` 应解析为 `(a + b) = c`（赋值给非左值），应产生 compile-time error。但 ArkTS 编译器实际允许该表达式通过。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `a + b = c` | ✅ 编译通过 |
| Java | `a + b = c` | ❌ 编译错误: unexpected type |
| Swift | `a + b = c` | ❌ 编译错误: cannot assign to expression |

**分类**：D 类 — Spec 与实现不一致

**建议**：ArkTS 编译器应按照 spec 要求，在解析 `(a + b) = c` 时报赋值给非左值的错误。

### ID-02: 逻辑与后赋值给非左值编译通过 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_01_01_023_FAIL_LOGICAL_AND_ASSIGNMENT |
| **实测结果** | 编译通过（期望编译时错误） |
| **错误信息** | 无错误 |

**描述**：和 ID-01 类似，`&&` 优先级高于 `=`，`a && b = c` 应解析为 `(a && b) = c`（赋值给非左值），但编译器允许通过。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `a && b = c` | ✅ 编译通过 |
| Java | `a && b = c` | ❌ 编译错误 |

**分类**：D 类 — Spec 与实现不一致

**建议**：同 ID-01。

### ID-03: 非 boolean 类型作为三元条件编译通过 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_01_01_024_FAIL_TERNARY_PRECEDENCE_TYPE |
| **实测结果** | 编译通过（期望类型错误） |
| **错误信息** | 无错误 |

**描述**：`??` 优先级高于 `?:`，`x ?? 0 ? "ok" : "no"` 解析为 `(x ?? 0) ? "ok" : "no"`。`x ?? 0` 返回 `int`，使用 `int` 作为三元条件应产生类型错误（spec 要求三元条件为 boolean 类型），但 ArkTS 编译通过。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `(5 ?? 0) ? "ok" : "no"` | ✅ 编译通过，返回 "ok" |
| Java | `(5 != null ? 5 : 0) != 0 ? "ok" : "no"` | ❌ int 不能作为条件 |
| Swift | `(5 ?? 0) != 0 ? "ok" : "no"` | ❌ Int 不能作为条件 |

**分类**：D 类 — Spec 与实现不一致；也可能涉及 A 类（ArkTS 可能允许 truthy/falsy 转换）

**建议**：确认 ArkTS 是否允许数值类型隐式转换为 boolean 条件（类似 C 语言的 truthy/falsy 语义）。如果是设计选择，需要更新 spec；如果是不一致，需要修复编译器。

### ID-04: Swift 移位运算符优先级差异 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | cross_lang_verify |
| **实测结果** | `1+2<<3-1` 在 Swift = 16，在 ArkTS/Java = 12 |
| **差异类型** | 符合 ArkTS spec 的语言设计差异 |

**描述**：Swift 的移位运算符 `<<` `>>` 优先级高于加减运算符，而 ArkTS/Java 的加减优先级高于移位。

**跨语言对比**：

| 语言 | `1+2<<3-1` 结果 | 优先级规则 |
|------|-----------------|-----------|
| ArkTS | 12 | 加法(9) > 移位(11) |
| Java | 12 | 加法 > 移位 |
| Swift | 16 | 移位 > 加法（BitwiseShiftPrecedence > AdditionPrecedence） |

**分类**：符合 ArkTS spec 的语言设计差异（Swift 的设计不同）

### ID-05: 一元负号与幂运算的语法歧义 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_01_01_011_PASS_UNARY_MINUS_EXPONENTIATION |
| **实测结果** | `-3 ** 2` 要求加括号，`(-3) ** 2` 编译通过 |
| **差异类型** | 符合 ArkTS spec 的语言设计差异 |

**描述**：ArkTS 编译器要求对幂运算底数前的负号显式加括号。根据 spec，前置一元运算符（优先级 6）高于幂运算（优先级 7），但编译器要求 `(-3) ** 2` 而非 `-3 ** 2`。这与 Java/Swift 的 `-(3^2)` 语义不同。

**跨语言对比**：

| 语言 | 代码 | 结果 | 解析方式 |
|------|------|------|---------|
| ArkTS | `(-3) ** 2` | 9 | 一元负号高于 `**`，但需括号消歧义 |
| Java | `-Math.pow(3,2)` | -9 | 方法调用 > 一元负号 |
| Swift | `-pow(3.0,2.0)` | -9 | 函数调用 > 一元负号 |

**分类**：符合 ArkTS spec 的语言设计差异

### ID-06: `**` 幂运算返回 double 类型

| 字段 | 值 |
|------|-----|
| **复现用例** | 多个涉及 `**` 的用例 |
| **实测结果** | `2 ** 3 ** 2` 结果类型为 double，不能赋值给 int |
| **差异类型** | 符合 ArkTS spec 的语言设计差异 |

**描述**：`**` 运算符始终返回 `double` 类型，即使操作数和结果都是整数。这与 Java 的 `Math.pow()` 和 Swift 的 `pow()` 一致。但可能会导致整数上下文中的类型不匹配，需要显式类型转换。

**跨语言对比**：

| 语言 | 整数幂运算类型 |
|------|--------------|
| ArkTS | `double` |
| Java | `double` (Math.pow) |
| Swift | `Double` (pow) |

**分类**：符合 ArkTS spec 的语言设计差异
