# 7.36.1 Specifics of Constant Expressions Evaluation — ArkTS 与 Java/Swift/TS 行为差异及规范一致性报告

## 设计问题及差异清单

### ID-01: 整型常量求值精度 ⭐⭐⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_36_01_001, EXP_07_36_01_004 |
| **实测结果** | ArkTS 内部 bigint 任意精度求值；Java/Swift 固定精度可能溢出 |
| **差异类型** | 符合 ArkTS spec 的语言设计差异 |

**描述**：编译时常量表达式中整型运算的精度差异。ArkTS 在编译时常量折叠时使用内部 bigint 类型（任意精度），因此不会溢出。Java 的 `final` 常量和 Swift 的 `let` 常量均使用固定精度整数类型（int/long / Int），大整数常量表达式可能发生溢出。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `const x = 9223372036854775807 + 1` | ✅ 内部 bigint 求值得 9223372036854775808 |
| Java | `final long x = 9223372036854775807L + 1` | ❌ long 溢出，得 -9223372036854775808 |
| Swift | `let x = 9223372036854775807 + 1` | ❌ Int 溢出（编译时错误或运行时崩溃） |

**分类**：跨语言设计差异（ArkTS 的内部 bigint 避免了其他语言的溢出问题）

**建议**：保持内部 bigint 设计，编译时使用任意精度整数进行常量折叠是 ArkTS 相对于 Java/Swift 的一大优势，可避免常量表达式中的意外溢出。建议在规范中明确说明 bigint 内部类型仅在编译时使用，运行时不出现。

### ID-02: 幂运算语法 ⭐⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_36_01_002, EXP_07_36_01_005 |
| **实测结果** | ArkTS 原生 `**` 运算符，Java/Swift 需函数调用 |
| **差异类型** | 符合 ArkTS spec 的语言设计差异 |

**描述**：幂运算的语法差异。ArkTS 支持原生 `**` 幂运算符，结果始终为 double。Java 需通过 `Math.pow()` 函数调用实现，Swift 需通过 Foundation 的 `pow()` 函数调用实现。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `const x = 2 ** 3` | ✅ 原生运算符，结果 double: 8.0 |
| Java | `final double x = Math.pow(2, 3)` | ✅ 函数调用，结果 double: 8.0 |
| Swift | `let x = pow(2.0, 3.0)` | ✅ 函数调用（需 Foundation），结果 Double: 8.0 |

**分类**：跨语言设计差异（ArkTS 语法更简洁）

**建议**：保持 `**` 运算符，原生 `**` 运算符是 ArkTS 的语法优势，比函数调用更简洁。

### ID-03: `>>>` 无符号右移支持 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_36_01_001, EXP_07_36_01_004 |
| **实测结果** | ArkTS 和 Java 支持 `>>>`，Swift 不支持 |
| **差异类型** | 符合 ArkTS spec 的语言设计差异 |

**描述**：无符号右移运算符的支持差异。ArkTS 和 Java 均支持 `>>>` 无符号右移运算符，Swift 不支持该运算符，需用 `>>` 代替（正数时行为等价）。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `256 >>> 2` | ✅ 64 |
| Java | `256 >>> 2` | ✅ 64 |
| Swift | `256 >>> 2` | ❌ 编译错误，不支持 |

**分类**：跨语言设计差异（轻微差异，正数时 `>>` 等价）

**建议**：无改进建议，当前设计合理。
