# 7.36.1 Specifics of Constant Expressions Evaluation — 三语言对比报告

## 1. 概览

| 维度 | ArkTS | Java | Swift |
|:----:|:-----:|:----:|:-----:|
| 常量声明 | `const` 关键字 | `final` 关键字 | `let` 关键字 |
| 整型常量折叠 | 内部使用 bigint（任意精度） | 使用 int/long（固定精度，可能溢出） | 使用 Int（固定精度，可能溢出） |
| double 提升 | `double + int` → `double` | `double + int` → `double` | `Double + Int` → `Double` |
| float 提升 | `float + int` → `float` | `float + int` → `float` | `Float + Int` → `Float` |
| 幂运算 | `**` 运算符，结果始终 double | `Math.pow()` 函数，结果 double | `pow()` 函数（Foundation），结果 double |
| 常量表达式范围 | 数值、字符串、关系、逻辑、移位、位运算、三元 | 数值、字符串、关系、逻辑、移位、位运算、三元 | 数值、字符串、关系、逻辑、移位、位运算、三元 |
| `>>>` 无符号右移 | ✅ 支持 | ✅ 支持 | ❌ 不支持 |

## 2. 章节对应关系

| ArkTS 概念 | Java | Swift |
|-----------|------|-------|
| `const c = 10 + 20` | `final int c = 10 + 20` | `let c = 10 + 20` |
| `const d: double = 3.0 + 5` | `final double d = 3.0 + 5` | `let d: Double = 3.0 + 5` |
| `const p = 2 ** 3`（double） | `final double p = Math.pow(2, 3)` | `let p = pow(2.0, 3.0)` |
| 内部 bigint 类型 | 无（int/long 编译时折叠） | 无（Int 编译时折叠） |
| 混合常量表达式 `c>1 && c*2<8` | 同 ArkTS | 同 ArkTS |

## 3. 关键差异矩阵

| 差异点 | ArkTS | Java | Swift |
|--------|:-----:|:----:|:-----:|
| 整型常量求值精度 | ⭐⭐⭐ 内部 bigint 任意精度 | ⚠️ int/long 固定精度会溢出 | ⚠️ Int 固定精度会溢出 |
| 幂运算语法 | ⭐⭐⭐ 原生 `**` 运算符 | ⚠️ `Math.pow()` 函数调用 | ⚠️ `pow()` 函数调用 |
| `>>>` 无符号右移 | ✅ 支持 | ✅ 支持 | ❌ 不支持 |
| `~` 按位取反 | ✅ `int` 两补码 | ✅ `int` 两补码 | ✅ `Int` 两补码 |
| 常量表达式分类 | ✅ 完整的常量表达式定义 | ✅ 完整的常量表达式定义 | ✅ 编译时常量折叠 |
| float 类型支持 | ✅ `float` 类型 | ✅ `float` 类型（`f` 后缀） | ✅ `Float` 类型 |

### 关键差异详解

#### 差异 1: 整型常量求值精度 ⭐⭐⭐

| 语言 | 场景 | 行为 |
|:----:|------|------|
| ArkTS | `const x = 9223372036854775807 + 1` | ✅ 内部 bigint 求值得 9223372036854775808 |
| Java | `final long x = 9223372036854775807L + 1` | ❌ long 溢出，得 -9223372036854775808 |
| Swift | `let x = 9223372036854775807 + 1` | ❌ Int 溢出（编译时错误或运行时崩溃） |

ArkTS 使用内部 bigint 类型进行编译时常量折叠，避免溢出。Java 和 Swift 使用固定精度整数类型。

#### 差异 2: 幂运算语法 ⭐⭐

| 语言 | 代码 | 行为 |
|:----:|------|------|
| ArkTS | `const x = 2 ** 3` | ✅ 原生运算符，结果 double: 8.0 |
| Java | `final double x = Math.pow(2, 3)` | ✅ 函数调用，结果 double: 8.0 |
| Swift | `let x = pow(2.0, 3.0)` | ✅ 函数调用（需 Foundation），结果 Double: 8.0 |

ArkTS 是唯一支持原生 `**` 幂运算符的语言。

#### 差异 3: `>>>` 无符号右移 ⭐

| 语言 | 代码 | 行为 |
|:----:|------|------|
| ArkTS | `256 >>> 2` → 64 | ✅ 支持 |
| Java | `256 >>> 2` → 64 | ✅ 支持 |
| Swift | `256 >>> 2` | ❌ 编译错误，不支持 |

Swift 不支持无符号右移运算符。

## 7. 核心结论

1. **ArkTS 的最大优势是内部 bigint 类型**：编译时常量折叠时使用任意精度整数，避免溢出。Java 和 Swift 使用固定精度类型，大整数常量表达式可能悄无声息地溢出。
2. **幂运算符 `**` 是 ArkTS 独有语法**：Java 和 Swift 都需通过函数调用 `Math.pow`/`pow` 实现，ArkTS 的原生运算符更简洁。
3. **double 提升和混合表达式语义三语言一致**：`double + int → double` 及混合常量表达式行为在各语言中表现相同。
4. **Swift 不支持 `>>>`**：无符号右移运算符在 Swift 中不可用，需用 `>>` 代替（正数时行为等价）。
5. **全部 6 个 ETS 用例通过**，Java 30/30 ✅，Swift 29/29 ✅，无 D 类异常。

## 8. ArkTS 设计建议

1. **保持内部 bigint 设计**：编译时使用任意精度整数进行常量折叠是 ArkTS 相对于 Java/Swift 的一大优势，可避免常量表达式中的意外溢出。
2. **保持 `**` 运算符**：原生 `**` 运算符是 ArkTS 的语法优势，比函数调用更简洁。
3. **文档建议**：建议在规范中明确说明 bigint 内部类型仅在编译时使用，运行时不出现，避免开发者混淆。
4. **无改进建议**：当前设计在常量表达式求值方面已经完善，无需改动。
