# 7.3 Literal — ArkTS 与 Java/Swift 行为差异及规范一致性报告

## 设计问题及差异清单

### ID-01: `.5` 前导小数点语法差异 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_03_002_PASS_FLOAT_BIGINT_LITERAL |
| **实测结果** | ArkTS 和 Java 支持 `.5`，Swift 编译错误 |
| **错误信息** | Swift: "'.5' is not a valid floating point literal; it must be written '0.5'" |
| **差异类型** | 语法级别的语言设计差异 |

**描述**：ArkTS 和 Java 支持浮点数字面量的前导小数点语法（如 `.5` 等价于 `0.5`），但 Swift 拒绝该写法并报编译错误，要求必须写作 `0.5`。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `.5` | ✅ 编译通过，值为 0.5 |
| Java | `.5` | ✅ 编译通过，值为 0.5 |
| Swift | `.5` | ❌ 编译错误，必须写 `0.5` |

**分类**：符合 ArkTS 规范的语言设计差异（Swift 语法更严格）

**建议**：无需修复。前导小数点语法在与 Java 保持一致的同时，提升了编码便利性。

### ID-02: 八进制字面量语法差异 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_03_001_PASS_INTEGER_LITERAL |
| **实测结果** | ArkTS/Swift 用 `0o` 前缀，Java 用 `0` 前缀 |
| **差异类型** | 语法级别的语言设计差异 |

**描述**：ArkTS 和 Swift 统一使用 `0o` 前缀表示八进制字面量（如 `0o77`），而 Java 延续传统 C 风格的 `0` 前缀（如 `077`）。

**跨语言对比**：

| 语言 | 八进制 77 | 行为 |
|------|-----------|------|
| ArkTS | `0o77` | ✅ 0o 前缀编译通过 |
| Java | `077` | ✅ 0 前缀（传统 C 风格） |
| Swift | `0o77` | ✅ 0o 前缀（同 ArkTS） |

**分类**：符合 ArkTS 规范的语言设计差异（ArkTS 选择现代风格）

**建议**：无需修复。`0o` 前缀比传统 `0` 前缀可读性更好，降低歧义。

### ID-03: ArkTS 独有字面量特性 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_03_002/003/004_PASS_* |
| **实测结果** | ArkTS 拥有 Java/Swift 均不支持的独占字面量特性 |
| **差异类型** | 符合 ArkTS 规范的设计差异 |

**描述**：ArkTS 独占以下 Java/Swift 均不支持的字面量特性：
- **bigint 字面量**：`153n`，Java 需 BigInteger 类，Swift 不支持大整数字面量
- **undefined 字面量**：`undefined` → void，Java/Swift 均无
- **自动整数类型推断**：按值范围自动选择 int/long，Java 需显式 L 后缀
- **单引号字符串**：`'Hello'` → string（Java 不支持）

**跨语言对比**：

| 特性 | ArkTS | Java | Swift |
|------|-------|------|-------|
| bigint 字面量 | ✅ `153n` → bigint | ❌ 需 BigInteger 类 | ❌ 无 |
| undefined 字面量 | ✅ `undefined` → void | ❌ 无 | ❌ 无 |
| 自动整数类型推断 | ✅ 按值范围自动 int/long | ❌ 需显式 L 后缀 | ❌ Int 统一类型 |
| 单引号字符串 | ✅ `'Hello'` → string | ❌ 仅双引号 | ✅ 支持 |

**分类**：符合 ArkTS 规范的设计差异（ArkTS 融合多种语言特性）

**建议**：无需修复。这些独占特性为 ArkTS 提供了更丰富的表达能力。

### ID-04: char 字面量实验性支持 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_03_005_PASS_UNDERSCORE_CHAR_LITERAL（已移除 char 行） |
| **实测结果** | `'A'` 被解析为字符串字面量类型 `"A"`，而非 char 类型 |
| **错误信息** | `Type '"A"' cannot be assigned to type 'Char'` |
| **差异类型** | 语言设计差异（char 为 ArkTS 实验特性） |

**描述**：ArkTS 的 `char` 类型为实验性特性。实测发现 `'A'` 在 ArkTS 中被解析为字符串字面量类型 `"A"`，而非 `char` 类型。因此 `char` 类型无法通过字面量直接赋值（`let c: char = 'A'` 产生类型错误）。

**跨语言对比**：

| 语言 | char 支持 |
|------|-----------|
| ArkTS | ⚠️ 实验性，不支持字面量直接赋值给 char |
| Java | ✅ `char c = 'A'` 原生支持 |
| Swift | ✅ `Character` 类型（通过 `Character("A")` 构造） |

**分类**：符合 ArkTS 规范的设计差异（char 为实验特性，尚未完全实现字面量支持）

**建议**：待 char 类型稳定后，考虑支持 `'A'` 字面量直接赋值给 char 类型。
