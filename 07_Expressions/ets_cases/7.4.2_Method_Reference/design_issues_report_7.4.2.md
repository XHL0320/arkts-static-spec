# 7.4.2 Method Reference — ArkTS 与 Java/Swift/TS 行为差异及规范一致性报告

## 设计问题及差异清单

### ID-01: 静态方法引用语法差异 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_02_001_PASS_STATIC_METHOD_REF |
| **实测结果** | `C.foo` 直接引用静态方法，编译通过 |
| **错误信息** | 无错误 |

**描述**：三语言对静态方法引用的支持方式和语法不同。ArkTS 语法最简洁。

**跨语言对比**：

| 语言 | 语法 | 说明 |
|------|------|------|
| ArkTS | `C.foo` | 直接引用静态方法，语法最简洁 |
| Java | `Class::foo` | `::` 运算符，需要函数式接口类型上下文 |
| Swift | 闭包包装 `{ Type.method($0, $1) }` | 无直接静态方法引用语法 |

**分类**：语法级别的语言设计差异，符合各自语言设计

**建议**：无缺陷，ArkTS 的 `C.foo` 语法简洁明了。

### ID-02: 显式重载语法（ArkTS 独有）⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_02_005_PASS_INDIVIDUAL_OVERLOAD_METHOD_REF / EXP_07_04_02_008_FAIL_EXPLICIT_OVERLOAD_METHOD_REF |
| **实测结果** | `overload foo { foo1, foo2 }` 中 `c.foo` 作为引用报编译时错误，单方法引用 `foo2` 通过 |
| **错误信息** | 编译时错误（使用重载名称时） |

**描述**：ArkTS 支持 `overload foo { foo1, foo2 }` 显式重载声明，Java/Swift 均无此概念。方法重载在 Java/Swift 中永远是隐式的。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `overload foo { foo1, foo2 }` → `c.foo` | ❌ 编译时错误 |
| ArkTS | `c.foo2` | ✅ 引用单个重载方法 |
| Java | N/A | ❌ 无显式重载语法 |
| Swift | N/A | ❌ 无显式重载语法 |

**分类**：语法级别的语言设计差异（ArkTS 独有特性），符合 ArkTS 规范的设计差异

**建议**：无缺陷，显式重载是 ArkTS 为代码可读性增加的显式声明机制。

### ID-03: 泛型方法引用约束 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_02_004_PASS_GENERIC_METHOD_REF / EXP_07_04_02_006_FAIL_GENERIC_METHOD_REF_NO_ARGS |
| **实测结果** | 泛型方法引用 `gen<string>` 编译通过，`new C().gen` 无类型参数报编译时错误 |
| **错误信息** | 编译时错误（无类型参数时） |

**描述**：ArkTS 要求泛型方法引用必须携带显式类型参数，而 Java 和 Swift 允许通过类型推断或类型标注隐式确定泛型参数。

**跨语言对比**：

| 语言 | 代码 | 行为 |
|------|------|------|
| ArkTS | `new C().gen` | ❌ 编译时错误，必须 `gen<string>` |
| Java | `Function<String, String> f = Class::identity` | ✅ 目标类型推断 |
| Swift | `let f: (Int) -> Int = instance.identity` | ✅ 类型标注推断 |

**分类**：编译时安全检查差异，符合 ArkTS 规范的设计差异

**建议**：当前设计合理，编译时强制类型参数提高了代码安全性。

### ID-04: 重载方法名引用处理策略 ⭐

| 字段 | 值 |
|------|-----|
| **复现用例** | EXP_07_04_02_007_FAIL_IMPLICIT_OVERLOADED_METHOD_REF |
| **实测结果** | 隐式重载方法名作为引用报编译时错误 |
| **错误信息** | 编译时错误 |

**描述**：三语言都处理重载方法名作为引用时的歧义，但策略不同。ArkTS 最严格，直接禁止引用重载方法名。

**跨语言对比**：

| 语言 | 消除策略 |
|------|---------|
| ArkTS | 直接禁止引用重载方法名（最严格） |
| Java | 通过目标函数式接口类型自动消除歧义 |
| Swift | 通过显式类型标注消除歧义 |

**分类**：编译时重载解析策略差异，符合各自语言设计

**建议**：ArkTS 策略最安全，不推荐放宽。
